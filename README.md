# Система автоматической сверки актов взаиморасчётов

Прототип системы для автоматического сопоставления хозяйственных операций
из актов сверки контрагентов (Сторона А и Сторона Б). Разработано в рамках
тестового задания на позицию **AI Architect / ML Engineer**.

Фокус решения — **детерминированность, объяснимость (explainability) и
защита от ложных срабатываний**: сопоставление не «угадывает» первого
подходящего кандидата, а оценивает пары независимо по номеру/дате/сумме
и явно помечает неоднозначность вместо тихого выбора.

## Соответствие постановке задачи

| № | Требование ТЗ | Реализация | Где в коде |
|---|---|---|---|
| 1 | Принимает два акта сверки (PDF/скан, при желании XLSX/CSV) | Два пути: (а) уже готовые TSV — для приложенных к заданию сканов; (б) **новый PDF напрямую** — автоматический OCR через Claude vision API одной командой | `run.py --side-a-pdf/--side-b-pdf`, `src/ocr_ingest.py` |
| 2 | Распознаёт операции, приводит к единой структуре | Карточка операции: тип, номер (сырой/нормализованный/ядро), дата документа (не дата проводки), сумма, направление | `src/models.py: OperationCard`, `src/parsing.py` |
| 3 | Сопоставляет при разных наименованиях документов | Словарь типов сводит «Акт», «Продажа», «Приход», «УПД», «Накладная» к общей группе «Отгрузка» (аналогично «Платёжное поручение» / «Оплата» → «Оплата»); сравнение идёт по группе + номеру + дате + сумме, а не по тексту наименования | `src/parsing.py: DOC_TYPE_GROUPS` |
| 4 | Выявляет: отсутствие у стороны / расхождения по сумме и дате / дубли / неоднозначные совпадения | Все четыре реализованы отдельными статусами: «Отсутствует у Покупателя/Поставщика», «Расхождение по сумме» / «по дате» / «по сумме и дате одновременно», «Возможный дубль», «Неоднозначное совпадение — ручная проверка» | `src/matching.py: Matcher.run()` |
| 5 | Итоговый Excel-отчёт с объяснением каждого результата | 8 листов, у каждой строки сопоставления — текстовое объяснение (какие признаки совпали/разошлись) | `src/report.py` |
| 6 | Сравнение с эталоном, метрики качества | Precision/Recall/F1 по списку расхождений + сравнение сумм, выводится в консоль и в отчёт | `src/metrics.py` |

## Быстрый старт

### Вариант А — на приложенных к заданию сканах (уже распознаны, без API)

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python run.py
```

Отчёт — `output/Отчёт_сверки.xlsx`. В консоль выводятся Precision/Recall/F1
относительно эталона. Работает офлайн: `data/raw/*.tsv` — уже распознанные
строки обоих актов (как именно получены — раздел «OCR» в
[ТЕХНИЧЕСКИЙ_РАЗБОР.md](ТЕХНИЧЕСКИЙ_РАЗБОР.md)).

### Вариант Б — на двух НОВЫХ PDF-сканах

```bash
pip install anthropic
export ANTHROPIC_API_KEY=...
python run.py --side-a-pdf path/to/scan_a.pdf --side-b-pdf path/to/scan_b.pdf
```

Страницы распознаются постранично через Claude vision API, результат
кэшируется в `data/raw/*.tsv`, дальше — тот же детерминированный пайплайн.
Стоимость/приватность вызова — докстринг `src/ocr_ingest.py`. Офлайн-
альтернатива без API — `src/ocr/tesseract_provider.py`; без пакета
`tesseract-ocr-rus` (недоступен в среде разработки — нет сети до apt)
точность на кириллице не проверена, см. тех.разбор.

### Прочие параметры

```bash
python run.py --mode strict            # или flexible (ТЗ п.6.5)
python run.py --side-a a.tsv --side-b b.tsv --reference ref.xlsx --output out.xlsx
python -m unittest discover -s tests -v   # 19 тестов, без pytest
```

## Архитектура решения

Модульно, по слоям, каждый заменяем независимо от остальных:

1. **Извлечение (`src/ocr/`, `src/ocr_ingest.py`)** — интерфейс
   `OCRProvider` с двумя реализациями: `TesseractProvider` (офлайн) и
   приём через Claude vision API. Скан → построчный текст.
2. **Нормализация (`src/parsing.py`, `src/models.py`)** — текст строки →
   `OperationCard`: тип документа по словарю, номер (raw/normalized/
   prefix/core — АБ-172 и РН-172 не путаются, т.к. префикс не отбрасывается
   молча), дата документа (приоритет над датой проводки), сумма+направление.
3. **Сопоставление (`src/matching.py`)** — балльная система (0-100) по
   трём независимым признакам вместо каскада `if/else`: пары сортируются
   по баллам, при равных баллах у нескольких кандидатов решение не
   принимается автоматически — статус «Ручная проверка». Отдельно —
   ограниченный комбинаторный поиск разбивки суммы на N строк (уровень F,
   `itertools.combinations` с лимитом глубины).
4. **Отчётность (`src/report.py`)** — Excel в 8 листов, каждое решение —
   с текстовым объяснением, откуда взялся статус.

## Структура проекта

```text
.
├── data/
│   ├── raw/                  # TSV с распознанными строками (или получаются из PDF, см. выше)
│   └── reference/            # эталон для расчёта метрик
├── output/                   # сгенерированные отчёты
├── src/
│   ├── ocr/                  # OCRProvider: tesseract_provider.py, base.py
│   ├── ocr_ingest.py         # новый PDF -> TSV через Claude vision API
│   ├── models.py             # OperationCard
│   ├── parsing.py            # тип/номер/дата/сумма
│   ├── matching.py           # скоринговый движок, уровни A-F
│   ├── metrics.py            # Precision/Recall/F1 против эталона
│   ├── pipeline.py           # оркестрация
│   └── report.py             # Excel-отчёт
├── tests/                    # unittest, 19 тестов
├── run.py                    # CLI
└── ТЕХНИЧЕСКИЙ_РАЗБОР.md      # архитектурные решения, разбор сложных случаев, ограничения
```

## Известные ограничения

- **`Инструкция.md`** — ТЗ называет её основным источником спецификации;
  в приложенных к заданию файлах её не было. Реализация — по «обязательному
  минимуму», описанному в самом ТЗ.
- **Качество исходных данных.** `data/raw/*.tsv` для приложенных сканов
  получены не классическим Tesseract (в среде разработки не было сети до
  `apt`/GitHub для установки `tesseract-ocr-rus`; на английской модели OCR
  давал даты вида «20.07.2001» вместо 2021 года — см. тех.разбор), а через
  визуальное считывание. Контрольная сверка с «Оборотами за период» из
  самих актов показала **недостачу ~3-7% строк** — источник большей части
  расхождения с идеальным F1.
- Уровень F (разбивка суммы на N строк) реализован и покрыт тестами, но
  на приложенных данных ни разу не сработал — проверен только синтетически.

## Проверка работоспособности

1. `python -m unittest discover -s tests -v` — все тесты зелёные.
2. `python run.py` — детерминирован: на приложенных данных всегда
   `Сторона А: 919 строк | Сторона Б: 1182 строк`, `F1=0.824`.
3. Лист «Сводка» в отчёте — сверить с таблицей выше.
4. Несколько строк с листа «Не найдено у Покупателя» — сверить вручную со
   сканом; если документ в скане есть, а в TSV нет — это пробел
   транскрипции, а не алгоритма (см. «Известные ограничения»).
